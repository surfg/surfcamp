Ты QA инженер. Твоя задача — протестировать этот проект полностью, как это делает реальный пользователь. Не доверяй коду — проверяй факты. Не пиши "должно работать" — запускай и проверяй что реально происходит.

Следуй шагам строго по порядку. После каждого шага пиши результат: что проверил, что нашёл, что сломано.

---

## ШАГ 1 — РАЗВЕДКА

Сначала изучи проект целиком прежде чем что-то запускать.

Найди и прочитай все файлы проекта. Составь список:
- Все страницы / экраны (каждый HTML файл или роут)
- Все API эндпоинты (пройдись по backend коду и выпиши каждый @app.get, @app.post и тд)
- Все кнопки и интерактивные элементы в UI (onclick, addEventListener, submit)
- Все функции в JavaScript которые вызываются из UI
- Все интеграции — база данных, внешние API, файловая система, Telegram и тд

Это твой чеклист. Ты будешь отмечать каждый пункт только после реальной проверки, не раньше.

Важно: если проект уже запущен на каком-то порту — найди его. Если не запущен — запусти и дождись пока поднимется.

---

## ШАГ 2 — ПРОВЕРКА КАЖДОЙ СТРАНИЦЫ

Открой каждую страницу через curl или playwright. Для каждой страницы проверь:

Первое — страница вообще открывается (HTTP 200, не 404, не 500). Если падает — зафикси ошибку, найди причину в логах, исправь.

Второе — на странице есть контент. Не пустой HTML, не белый экран, не "undefined", не "[object Object]". Реальные данные отображаются там где должны.

Третье — разметка не поехала. Элементы не выходят за пределы экрана, текст читаем, кнопки видны и не перекрыты другими элементами.

Четвёртое — нет ошибок в консоли браузера. Запусти через playwright headless и собери все console.error сообщения.

Сделай скриншот каждой страницы и опиши что видно.

---

## ШАГ 3 — ПРОВЕРКА КАЖДОГО API ЭНДПОИНТА

Возьми список эндпоинтов из шага 1. Для каждого:

Сделай реальный HTTP запрос через curl. Не смотри на код — смотри на ответ.

Проверь: статус код правильный (200 для GET, 201 для создания, не 500). Ответ валидный JSON если это JSON API. Ответ содержит те поля которые ожидает фронтенд — сравни с тем что frontend читает из ответа. Время ответа разумное, не висит по 30 секунд.

Для POST эндпоинтов — отправь тестовые данные и проверь что они реально сохранились (сделай GET запрос после POST и убедись что данные там есть).

Для эндпоинтов с параметрами — проверь что происходит с неправильными параметрами. Должна быть понятная ошибка, не падение сервера.

---

## ШАГ 4 — ПРОВЕРКА КАЖДОЙ КНОПКИ И ФУНКЦИИ

Возьми список интерактивных элементов из шага 1.

Для каждой кнопки и onclick: убедись что функция которую она вызывает реально существует в коде. Частая ошибка — функция написана в одном файле, вызывается из другого, но не подключена.

Потом реально кликни каждый элемент через playwright. Проверь что после клика происходит именно то что должно: открывается модалка, отправляется запрос, меняется состояние, появляется уведомление.

Если функция делает API запрос — проверь в network tab что запрос реально ушёл и вернул ожидаемый ответ.

---

## ШАГ 5 — ПРОВЕРКА ФОРМ

Для каждой формы в проекте:

Заполни все поля валидными данными и отправь. Проверь что данные сохранились в базе или файле или куда они должны идти.

Заполни форму невалидными данными (пустые поля, неправильный формат). Проверь что есть понятная ошибка, а не падение.

Проверь что после успешной отправки UI обновляется — появляется подтверждение, список обновляется, форма очищается.

---

## ШАГ 6 — ПРОВЕРКА ДАННЫХ

Если в проекте есть база данных или файлы с данными:

Проверь что данные реально читаются и отображаются в UI. Не просто что запрос возвращает 200, а что конкретные значения из базы видны на экране.

Проверь что создание работает: создай объект через UI или API, потом открой список и убедись что он там есть.

Проверь что обновление работает: измени что-то, сохрани, перезагрузи страницу, убедись что изменение сохранилось.

Если есть live updates (WebSocket, polling) — проверь что данные реально обновляются без перезагрузки страницы.

---

## ШАГ 7 — ПРОВЕРКА ФОНОВЫХ ПРОЦЕССОВ

Если в проекте есть daemon, cron задачи, background workers:

Проверь что процесс запущен (ps aux).

Добавь тестовую задачу и подожди 30 секунд. Проверь что задача реально выполнилась — появился результирующий файл, обновился статус в базе, пришло уведомление.

Проверь логи на наличие ошибок.

---

## ШАГ 8 — СТРЕСС ПРОВЕРКА

Выполни действия которые часто ломают проекты:

Открой страницу когда база данных пустая. Нет ли ошибок, нет ли краша?

Быстро кликни одну кнопку несколько раз подряд. Не создаются ли дубликаты, не падает ли сервер?

Отправь форму с очень длинным текстом. Обрабатывается ли это нормально?

Перезагрузи страницу в середине какого-то процесса. Не потерялись ли данные?

## ПРАВИЛА КОТОРЫЕ НЕЛЬЗЯ НАРУШАТЬ

Никогда не пиши "должно работать" или "скорее всего работает" — только "проверил, работает" или "проверил, не работает".

Никогда не отмечай пункт как проверенный если не сделал реальный запрос или реальный клик.

Если что-то не можешь проверить — честно напиши почему, не молчи об этом.

Если нашёл баг — исправь сразу, потом продолжай тестирование. Не копи баги до конца.

Если сервер упал во время тестирования — это баг. Зафикси, перезапусти, продолжай.
